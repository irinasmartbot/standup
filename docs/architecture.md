# Moscow StandUp Show — Архитектура проекта

## Общая схема системы

```
Google Таблица (Афиша)
        ↓
    Telegram Бот  ←→  База данных (SQLite → PostgreSQL)
        ↓
    Пользователь
```

**Текущее состояние:** на сервере мероприятия синхронизируются из Google Таблицы в PostgreSQL, бот читает события из PostgreSQL.  
**Локальная разработка:** при необходимости может читать мероприятия из Google Таблицы напрямую.  
**В будущем:** Google Таблица заменяется на Админ-панель для менеджера.

## Форматы шоу

### StandUp Проверка материала
- Вход бесплатный
- 5–7 комиков по 10–15 минут
- Воронка бронирования реализована полностью
- Подробнее: [flows/proverka_materiala.md](flows/proverka_materiala.md)

### StandUp BEST
- Билеты от 500 рублей
- Показывает даты и карточки событий из `events`
- Кнопка ведёт на ссылку оплаты из Google Sheets
- Бронь в `bookings` не создаётся

### Розыгрыш
- Вход: `?start=standup_rozygr`
- Ветки: билет за пост / билет за отзыв → модерация скрина → подписка на канал → даты BEST (кроме сегодня) → бронь на 1 гостя (`format=rozygrysh`)
- Свои напоминания/аннуляция (отдельно от проверки материала)
- Флаг `users.rozygrysh_used`; сброс при отмене/аннуляции
- Deep link друга на платный BEST: `?start=afisha_plat`
- Нужны env: `MODERATION_CHAT_ID`, `CHANNEL_USERNAME`

### Форматы в плане
- 1+1
- Мастер-классы (с оплатой и проверкой чека)
- Хит-лото
- Сертификаты (актуальность под вопросом)

Рабочий SSH-пользователь для деплоя: `standup@31.128.47.4`. Вход настроен по локальному SSH-ключу, пароль и приватный ключ в репозиторий не добавляются.

Firewall (`ufw`) включён. Разрешены входящие порты: `22/tcp` для SSH, `80/tcp` для HTTP, `443/tcp` для HTTPS.

PostgreSQL проверен на VPS: порт `5432` слушает только локальный адрес `127.0.0.1`.

Код проекта скачан на VPS в `/home/standup/app`, активная ветка на сервере: `dev`.

Серверный бот запущен через `systemd`-службу `standup-bot.service`; служба включена в автозапуск и проверена в статусе `active (running)`.

Автодеплой настроен через GitHub Actions: каждый `push` в ветку `dev` обновляет `/home/standup/app`, устанавливает зависимости и перезапускает `standup-bot`.

PostgreSQL backup настроен через `cron`: база `standup_db` сохраняется ежедневно в `/home/standup/backups/postgres`, backup'ы старше 14 дней удаляются автоматически.

Восстановление backup проверено на отдельной базе `standup_restore_test`: `pg_restore` успешно восстановил таблицы `events`, `users`, `bookings`.

Порядок ручного обновления системы и PostgreSQL описан в `docs/database.md`.

Синхронизация событий Google Sheets → PostgreSQL запущена на VPS через `systemd`-службу `standup-events-sync.service`; для текущей версии синхронизируется лист «Афиша бесплатников» в формат `proverka`.

Серверный бот читает события из PostgreSQL при `EVENTS_SOURCE=postgres` или при наличии `DATABASE_URL` без явного `EVENTS_SOURCE`.

Серверный бот записывает новые брони и пользователей в PostgreSQL при `BOOKINGS_SOURCE=postgres` или при наличии `DATABASE_URL` без явного `BOOKINGS_SOURCE`. Тестовые записи из серверного `bookings.db` не переносились. Проверены создание брони, изменение количества гостей и отмена.

### Серверный запуск Telegram-бота

Серверная копия проекта находится в `/home/standup/app`. Рабочая ветка для настройки и тестирования сервера — `dev`; каждый `push` в `dev` запускает автодеплой через GitHub Actions.

Настройки сервера лежат в `/home/standup/app/.env`. Токены и пароли не хранятся в GitHub. Чтобы переключить тестового бота на другой Telegram-токен, нужно изменить `BOT_TOKEN` в `.env` и перезапустить сервис:

```bash
sudo systemctl restart standup-bot
sudo systemctl status standup-bot
```

Посмотреть последние логи бота:

```bash
sudo journalctl -u standup-bot -n 100 --no-pager
```
