# База данных — схема таблиц

## Общие правила

- Записи **никогда не удаляются** — только меняется статус
- Удаление доступно только администратору и только намеренно
- Менеджер может архивировать, но не удалять

---

## Таблица `events` — мероприятия

Все форматы шоу хранятся в одной таблице, разделяются по полю `format`.

| Поле | Тип | Описание |
|---|---|---|
| id | INTEGER | Уникальный номер |
| format | TEXT | Формат шоу (см. ниже) |
| date | TEXT | Дата (ДД.ММ.ГГГГ) |
| weekday | TEXT | День недели |
| time | TEXT | Время начала |
| address | TEXT | Полный адрес площадки |
| location | TEXT | Короткое название (Escobar) |
| description | TEXT | Описание мероприятия |
| image_url | TEXT | Ссылка на афишу |
| price | INTEGER | Стоимость (0 = бесплатно) |
| payment_url | TEXT | Ссылка на оплату (для платных) |
| host | TEXT | Ведущий (для мастер-классов) |
| max_seats | INTEGER | Максимум мест |
| status | TEXT | active / past |
| updated_at | TEXT | Когда последний раз обновлялось |

### Значения поля `format`

| Значение | Что это | Бесплатно |
|---|---|---|
| `proverka` | Проверка материала | Да |
| `1plus1` | 1+1 | Да |
| `best` | StandUp BEST | Нет |
| `masterclass` | Мастер-класс | Нет |
| `hitloto` | Хит-лото | Нет |

### Статус мероприятия
- Присваивается **автоматически** по дате — не нужно менять вручную
- `active` — дата >= сегодня
- `past` — дата < сегодня
- Менеджер может вручную снять мероприятие не удаляя его

### Синхронизация с Google Таблицей
Пока нет админ-панели — раз в час скрипт читает каждый лист таблицы и обновляет базу:

| Лист Google Таблицы | format в БД |
|---|---|
| Проверка материала | proverka |
| 1+1 | 1plus1 |
| StandUp BEST | best |
| Хит-лото | hitloto |
| Мастер-классы | masterclass |

Каждый лист читается отдельно — данные не смешиваются.

## Таблица `users` — пользователи

Запись создаётся когда пользователь оставил номер телефона в процессе бронирования бесплатной ветки. По платным веткам (BEST, мастер-классы, хит-лото) пользователи в таблицу не попадают.

| Поле | Тип | Описание |
|---|---|---|
| id | INTEGER | Уникальный номер |
| telegram_id | INTEGER | ID в Telegram (главный идентификатор) |
| vk_id | INTEGER | ID ВКонтакте (заполняется при подключении VK-бота) |
| username | TEXT | Никнейм в Telegram |
| name | TEXT | Имя (из профиля или введённое вручную) |
| phone | TEXT | Номер телефона |
| source | TEXT | Источник: telegram / vkontakte |
| rozygrysh_used | INTEGER | Использовал розыгрыш: 0 = нет, 1 = да |
| created_at | TEXT | Дата первого обращения |
| last_active_at | TEXT | Дата последней активности |

### Логика `rozygrysh_used`

| Значение | Когда |
|---|---|
| `0` (не использовал) | Ещё не участвовал |
| `0` (не использовал) | Участвовал но отменил бронь — шанс возвращается |
| `1` (использовал) | Получил билет и не отменил — даже если не пришёл |

Менеджер может вручную сбросить значение на `0` через админ-панель (если клиент не пришёл по уважительной причине).

### Главный идентификатор — `telegram_id`
- Уникален для каждого человека, никогда не меняется
- Когда человек пишет боту — ищем его по `telegram_id`
- Есть в базе → берём существующего пользователя
- Нет в базе → создаём нового

## Таблица `bookings` — брони

Содержит только брони по бесплатным веткам и розыгрышам. По платным веткам брони не фиксируются.

**Фиксируются:**
- Проверка материала
- 1+1
- Розыгрыш

**НЕ фиксируются:**
- StandUp BEST
- Мастер-классы
- Хит-лото

| Поле | Тип | Описание |
|---|---|---|
| id | INTEGER | Уникальный номер брони |
| user_id | INTEGER | Ссылка на users.id |
| event_id | INTEGER | Ссылка на events.id |
| guests | INTEGER | Количество гостей (1–4) |
| format | TEXT | proverka / 1plus1 / rozygrysh |
| source | TEXT | telegram / vkontakte |
| status | TEXT | Статус брони (см. ниже) |
| created_at | TEXT | Когда создана |
| confirmed_at | TEXT | Когда получил билет |
| annulled_at | TEXT | Когда аннулирована системой |
| reminder_24h_sent | INTEGER | Напоминание за сутки отправлено (0/1) |
| reminder_day_sent | INTEGER | Напоминание в день шоу отправлено (0/1) |

### Статусы брони

| Статус | Когда возникает |
|---|---|
| Забронировано | Человек прошёл все шаги, бронь создана |
| Подтверждено | Нажал «Получить билет» |
| Отменено | Сам нажал «Отменить бронь» |
| Аннулировано | Не подтвердил вовремя — система отменила автоматически |

### Розыгрыш — особые правила
- Человек может участвовать **только один раз** (проверяется по `users.rozygrysh_used`)
- При выборе даты показываются только мероприятия BEST с датой **строго позже сегодняшнего дня**
- Бронь по розыгрышу: `format = rozygrysh`, мероприятие остаётся с `format = best` в таблице events

---

## Связи между таблицами

```
users (id) ←── bookings (user_id)
events (id) ←── bookings (event_id)
```

Один пользователь может иметь много броней.  
Одно мероприятие может иметь много броней.  
Каждая бронь привязана к одному пользователю и одному мероприятию.

## Импорт начальных данных

При запуске новой системы:

1. **Из Salebot CSV** → таблица `users` (только пользователи с номером телефона, за последние 12 месяцев)
2. **Из Google Таблицы** → таблица `bookings` (только активные брони на будущие мероприятия)
3. Сопоставление по номеру телефона — если телефон уже есть в `users`, дубль не создаётся
