# База данных — схема таблиц

## Общие правила

- Записи **никогда не удаляются** — только меняется статус
- Удаление доступно только администратору и только намеренно
- Менеджер может архивировать, но не удалять
- Серверная PostgreSQL-схема хранится в `bot/db/schema_postgres.sql`

---

## Таблица `events` — мероприятия

Все форматы шоу хранятся в одной таблице, разделяются по полю `format`.

| Поле | Тип | Описание |
|---|---|---|
| id | INTEGER | Уникальный номер |
| format | TEXT | Формат шоу (см. ниже) |
| date | TEXT | Дата (ДД.ММ.ГГГГ) |
| weekday | TEXT | День недели |
| time | TEXT | Время начала |
| address | TEXT | Полный адрес площадки |
| location | TEXT | Короткое название (Escobar) |
| description | TEXT | Описание мероприятия |
| image_url | TEXT | Ссылка на афишу |
| price | INTEGER | Стоимость (0 = бесплатно) |
| payment_url | TEXT | Ссылка на оплату (для платных) |
| host | TEXT | Ведущий (для мастер-классов) |
| max_seats | INTEGER | Максимум мест |
| status | TEXT | active / past |
| updated_at | TEXT | Когда последний раз обновлялось |

### Значения поля `format`

| Значение | Что это | Бесплатно |
|---|---|---|
| `proverka` | Проверка материала | Да |
| `1plus1` | 1+1 | Да |
| `best` | StandUp BEST | Нет |
| `masterclass` | Мастер-класс | Нет |
| `hitloto` | Хит-лото | Нет |

### Статус мероприятия
- Присваивается **автоматически** по дате — не нужно менять вручную
- `active` — дата >= сегодня
- `past` — дата < сегодня
- Менеджер может вручную снять мероприятие не удаляя его

### Синхронизация с Google Таблицей
Пока нет админ-панели — раз в час скрипт читает каждый лист таблицы и обновляет базу:

| Лист Google Таблицы | format в БД |
|---|---|
| Проверка материала | proverka |
| 1+1 | 1plus1 |
| StandUp BEST | best |
| Хит-лото | hitloto |
| Мастер-классы | masterclass |

Каждый лист читается отдельно — данные не смешиваются.

## Таблица `users` — пользователи

Запись создаётся когда пользователь оставил номер телефона в процессе бронирования бесплатной ветки. По платным веткам (BEST, мастер-классы, хит-лото) пользователи в таблицу не попадают.

| Поле | Тип | Описание |
|---|---|---|
| id | INTEGER | Уникальный номер |
| telegram_id | INTEGER | ID в Telegram (главный идентификатор) |
| vk_id | INTEGER | ID ВКонтакте (заполняется при подключении VK-бота) |
| username | TEXT | Никнейм в Telegram |
| name | TEXT | Имя (из профиля или введённое вручную) |
| phone | TEXT | Номер телефона |
| source | TEXT | Источник: telegram / vkontakte |
| rozygrysh_used | INTEGER | Использовал розыгрыш: 0 = нет, 1 = да |
| created_at | TEXT | Дата первого обращения |
| last_active_at | TEXT | Дата последней активности |

### Логика `rozygrysh_used`

| Значение | Когда |
|---|---|
| `0` (не использовал) | Ещё не участвовал |
| `0` (не использовал) | Участвовал но отменил бронь — шанс возвращается |
| `1` (использовал) | Получил билет и не отменил — даже если не пришёл |

Менеджер может вручную сбросить значение на `0` через админ-панель (если клиент не пришёл по уважительной причине).

### Главный идентификатор — `telegram_id`
- Уникален для каждого человека, никогда не меняется
- Когда человек пишет боту — ищем его по `telegram_id`
- Есть в базе → берём существующего пользователя
- Нет в базе → создаём нового

## Таблица `bookings` — брони

Содержит только брони по бесплатным веткам и розыгрышам. По платным веткам брони не фиксируются.

**Фиксируются:**
- Проверка материала
- 1+1
- Розыгрыш

**НЕ фиксируются:**
- StandUp BEST
- Мастер-классы
- Хит-лото

| Поле | Тип | Описание |
|---|---|---|
| id | INTEGER | Уникальный номер брони |
| user_id | INTEGER | Ссылка на users.id |
| event_id | INTEGER | Ссылка на events.id |
| guests | INTEGER | Количество гостей (1–4) |
| format | TEXT | proverka / 1plus1 / rozygrysh |
| source | TEXT | telegram / vkontakte |
| status | TEXT | Статус брони (см. ниже) |
| created_at | TEXT | Когда создана |
| confirmed_at | TEXT | Когда получил билет |
| annulled_at | TEXT | Когда аннулирована системой |
| reminder_24h_sent | INTEGER | Напоминание за сутки отправлено (0/1) |
| reminder_day_sent | INTEGER | Напоминание в день шоу отправлено (0/1) |

### Статусы брони

| Статус | Когда возникает |
|---|---|
| Забронировано | Человек прошёл все шаги, бронь создана |
| Подтверждено | Нажал «Получить билет» |
| Отменено | Сам нажал «Отменить бронь» |
| Аннулировано | Не подтвердил вовремя — система отменила автоматически |

### Розыгрыш — особые правила
- Человек может участвовать **только один раз** (проверяется по `users.rozygrysh_used`)
- При выборе даты показываются только мероприятия BEST с датой **строго позже сегодняшнего дня**
- Бронь по розыгрышу: `format = rozygrysh`, мероприятие остаётся с `format = best` в таблице events

## Связи между таблицами

```
users (id) ←── bookings (user_id)
events (id) ←── bookings (event_id)
```

Один пользователь может иметь много броней.  
Одно мероприятие может иметь много броней.  
Каждая бронь привязана к одному пользователю и одному мероприятию.

## Импорт начальных данных

При запуске новой системы:

1. **Из Salebot CSV** → таблица `users` (только пользователи с номером телефона, за последние 12 месяцев)
2. **Из Google Таблицы** → таблица `bookings` (только активные брони на будущие мероприятия)
3. Сопоставление по номеру телефона — если телефон уже есть в `users`, дубль не создаётся

### Порядок запуска на сервере

Минимальный порядок для текущей версии, где полноценно работает ветка «Проверка материала»:

1. Применить PostgreSQL-схему:

```bash
psql -d standup_db -f bot/db/schema_postgres.sql
```

2. Заполнить серверный `.env`, включая `DATABASE_URL`.

3. Импортировать актуальные мероприятия «Проверки материала»:

```bash
python scripts/import_events_from_sheets.py
```

4. При необходимости перенести активные будущие брони из локальной SQLite-базы:

```bash
python scripts/migrate_bookings_from_sqlite.py
```

5. Запустить синхронизацию мероприятий:

```bash
python scripts/sync_events_hourly.py
```

На продакшен-сервере синхронизация должна запускаться не вручную, а через `systemd` на Шаге 3.

### Импорт мероприятий из Google Sheets

Скрипт `scripts/import_events_from_sheets.py` импортирует актуальные будущие мероприятия из опубликованного CSV Google Sheets в таблицу `events`.

Сейчас скрипт покрывает текущий лист «Проверка материала». Остальные листы (`1+1`, BEST, МК, Хит-лото) добавляются позже вместе с реализацией соответствующих веток бота, когда будут понятны реальные колонки и ссылки CSV.

Перед запуском в `.env` должна быть переменная:

```env
DATABASE_URL=postgresql://standup_user:password@localhost:5432/standup_db
```

Запуск для текущего листа «Проверка материала»:

```bash
python scripts/import_events_from_sheets.py
```

Скрипт делает upsert по связке `format + event_date + event_time + location`: если мероприятие уже есть, оно обновляется.

### Синхронизация мероприятий раз в час

Скрипт `scripts/sync_events_hourly.py` запускает импорт мероприятий в цикле раз в час:

```bash
python scripts/sync_events_hourly.py
```

На VPS синхронизация запущена как `systemd`-служба `standup-events-sync.service`:

```bash
sudo systemctl status standup-events-sync
sudo journalctl -u standup-events-sync -n 100 --no-pager
```

Для текущей версии синхронизируется лист «Афиша бесплатников» в формат `proverka`.

### Перенос броней из SQLite

Скрипт `scripts/migrate_bookings_from_sqlite.py` переносит активные будущие брони из локальной SQLite-базы в PostgreSQL.

Запускать его нужно после импорта мероприятий, потому что каждая бронь привязывается к записи в таблице `events`.

```bash
python scripts/migrate_bookings_from_sqlite.py
```

По умолчанию скрипт берёт:

- SQLite-файл из `DB_PATH` или `bookings.db`
- PostgreSQL-подключение из `DATABASE_URL`

Переносятся только брони со статусом `booked` или `confirmed` на будущие даты. Если соответствующее мероприятие не найдено в `events`, бронь пропускается и выводится в отчёте.

### Отложенные импорты перед продакшен-запуском

Эти задачи обязательны перед переключением реального бота на PostgreSQL, но не блокируют текущую разработку и настройку сервера:

1. Импорт пользователей из Salebot CSV в `users`.
2. Импорт активных будущих броней из Google Sheets в `bookings`.
3. Сопоставление броней с пользователями по телефону.
4. Автоматическое создание пользователя, если в брони есть телефон, но пользователя ещё нет.

## Эксплуатация PostgreSQL на VPS

PostgreSQL используется как основная серверная база. Облачную БД Beget пока не используем: база работает на том же VPS, где запущен бот.

Первичная схема создаётся SQL-файлом:

Та же команда указана выше в порядке запуска.

Обязательные настройки перед запуском на реальных клиентах:

1. Разрешить подключение к PostgreSQL только локально внутри VPS.
2. Проверить, что порт PostgreSQL не открыт наружу через firewall.
3. Настроить ежедневный backup базы `standup_db`.
4. Хранить последние backup'ы минимум 7-14 дней.
5. Проверить восстановление backup'а на тестовой базе.
6. Описать ручной порядок обновления системы и PostgreSQL.

Локально для разработки можно продолжать использовать SQLite. Серверная версия должна работать с PostgreSQL.

### Backup PostgreSQL

Backup-скрипт хранится в `scripts/backup_postgres.sh`. На VPS он запускается каждый день через `cron`:

```bash
15 3 * * * /home/standup/app/scripts/backup_postgres.sh >> /home/standup/backups/postgres/backup.log 2>&1
```

Файлы backup'ов создаются в `/home/standup/backups/postgres` в формате custom dump (`pg_dump -F c`). Скрипт автоматически удаляет backup'ы старше 14 дней.

Ручной запуск:

```bash
/home/standup/app/scripts/backup_postgres.sh
```

После запуска нужно проверить, что появился `.dump`-файл не нулевого размера:

```bash
ls -lh /home/standup/backups/postgres
```

### Проверка восстановления backup

Восстановление проверяется на отдельной тестовой базе, рабочая база `standup_db` при этом не трогается:

```bash
sudo -u postgres dropdb --if-exists standup_restore_test
sudo -u postgres createdb -O standup_user standup_restore_test

cd /home/standup/app
set -a; source .env; set +a
RESTORE_URL="${DATABASE_URL/standup_db/standup_restore_test}"

pg_restore --no-owner --dbname="$RESTORE_URL" /home/standup/backups/postgres/<backup-file>.dump
psql "$RESTORE_URL" -c "\dt"
```

Успешная проверка: в тестовой базе видны таблицы `bookings`, `events`, `users`.

### Ручное обновление системы и PostgreSQL

Перед обновлением нужно убедиться, что есть свежий backup:

```bash
/home/standup/app/scripts/backup_postgres.sh
ls -lh /home/standup/backups/postgres
```

Обновление пакетов на VPS:

```bash
sudo apt update
sudo apt upgrade
```

После обновления проверить PostgreSQL и бота:

```bash
sudo systemctl status postgresql
sudo systemctl status standup-bot
```

Если система просит перезагрузку, выполнить её только после проверки backup'а:

```bash
sudo reboot
```

После перезагрузки снова проверить сервисы:

```bash
sudo systemctl status postgresql
sudo systemctl status standup-bot
```

Если бот не запущен:

```bash
sudo systemctl restart standup-bot
sudo journalctl -u standup-bot -n 100 --no-pager
```
